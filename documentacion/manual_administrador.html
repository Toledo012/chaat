01 Introducción

SEMAHN.TICKETS es el sistema institucional para solicitar, atender y documentar los servicios de tecnología (soporte, desarrollo, redes, mantenimiento) dentro de la SEMAHN. Cada solicitud se registra como un ticket, se asigna a un técnico del área de sistemas y, al concluir, se respalda con un formato de servicio oficial en PDF.

El sistema conecta a tres actores:

Departamentos

Levantan solicitudes de servicio (tickets) y les dan seguimiento hasta su cierre.

Sistemas / Técnicos

Toman los tickets, ejecutan el trabajo y llenan el formato de servicio correspondiente.

Administrador

Supervisa todo el proceso, gestiona usuarios, asigna tickets y controla catálogos y reportes.

Como administrador

Tienes acceso total: creas y administras cuentas, asignas tickets a técnicos, gestionas departamentos y materiales, y generas reportes globales de servicios.

04 Gestión de usuarios

Desde Usuarios administras las cuentas de acceso. Cada cuenta se compone de un usuario (datos de la persona: nombre, puesto, departamento, extensión, correo) y una cuenta (credenciales, rol y estado).

Crear un usuario

  1. Abre "Nuevo usuario" en la sección Usuarios.
  2. Captura los datos de la persona y asigna su departamento.
  3. Define el usuario y la contraseña de acceso.
  4. Selecciona el rol (Administrador, Usuario o Departamento).
  5. Guarda. La cuenta queda ACTIVA y lista para ingresar.

05 Gestión de departamentos

El catálogo de departamentos representa las áreas de la dependencia que pueden solicitar servicios. Cada ticket y cada formato quedan asociados a un departamento. La administración del catálogo (crear/editar) está reservada al Administrador.

  • Nombre — nombre del área (ej. "Recursos Humanos").
  • Descripción — referencia opcional.
  • Activo — solo los departamentos activos aparecen en los formularios.
Creación rápida

Los formularios de formato incluyen un botón "Nuevo" que permite dar de alta un departamento sin salir de la pantalla (función quick-store).

06 Gestión de tickets

El ticket es el corazón del sistema. Registra una solicitud de servicio desde que nace hasta que se cierra con su formato. Como administrador ves todos los tickets y controlas su ciclo de vida.

Ciclo de vida de un ticket

EstadoSignificadoQuién lo provoca
NUEVORecién creado, sin técnico asignado.Departamento / Admin / Técnico
ASIGNADOYa tiene un técnico responsable.Admin (asigna) o Técnico (toma)
EN PROCESOEl técnico inició la atención y se creó el servicio.Técnico (al dar "Completar")
EN ESPERAAtención pausada temporalmente.Admin
COMPLETADOFormato guardado; servicio documentado.Técnico (al guardar formato)
CANCELADOSolicitud anulada.Admin o dueño del ticket

Prioridades

BAJA   MEDIA   ALTA  — Los tickets creados por departamentos o técnicos toman media por defecto; el administrador puede fijar la prioridad al crear o editar.

Acciones del administrador

  1. Crear ticket — captura título, solicitante, descripción, prioridad, tipo de formato y departamento.
  2. Asignar — designa a un técnico responsable; el ticket pasa a asignado y se notifica por correo.
  3. Completar — deriva al llenado del formato de servicio correspondiente.
  4. Cancelar — anula la solicitud (no aplica a tickets ya completados).
  5. Editar — ajusta datos, prioridad o estado.

Folio del ticket

El folio se genera automáticamente con el patrón TCK-AAAAMMDD-SIGLA-NN, donde SIGLA son las iniciales del departamento y NN es el consecutivo del día para ese departamento. Ejemplo: TCK-20260703-RH-01.

07 Formatos y reportes

Al completar un ticket se genera un servicio que se documenta con un formato. Existen cuatro tipos de formato de servicio más el de recepción:

TipoUso
A Soporte y DesarrolloServicios de soporte técnico o desarrollo institucional.
B Equipo de cómputoMantenimiento y diagnóstico de equipos (con materiales).
C Redes / TelefoníaServicios de red y telefonía (con materiales).
D Resguardo / EntregaEntrega o resguardo de equipo.
R RecepciónFormato de recepción/conformidad del servicio.

Consulta, vista previa y PDF

  • Listado de formatos con filtros por tipo, técnico y rango de fechas.
  • Vista previa en pantalla antes de imprimir.
  • PDF individual por formato (motor DomPDF), con nombre FormatoX_<folio>.pdf.
  • Reporte general en PDF con estadísticas por técnico, tipo y mes.
Folio de servicio

Cada servicio recibe un folio institucional global con el patrón SEMAHN-<TIPO>-00001 (ej. SEMAHN-A-00042), independiente del folio del ticket.

08 Catálogo de materiales

Los formatos B y C permiten registrar los materiales utilizados en el servicio. El catálogo de materiales define los insumos disponibles.

  • Nombre del material y unidad sugerida.
  • Alta rápida desde el propio formato (quick-store).
  • Eliminación individual o múltiple.
Acceso

El catálogo de materiales está disponible para Administrador y Sistemas; queda bloqueado para el perfil Departamento.

12 Variables del sistema y su uso

Estas son las variables (campos) de control que rigen el comportamiento del sistema. Comprenderlas es esencial para dar mantenimiento.

Campo estado (tabla tickets)

Controla el ciclo de vida. Valores: nuevo, asignado, en_proceso, en_espera, completado, cancelado. Las transiciones las gobierna TicketService: un ticket completado o cancelado no puede asignarse, tomarse ni editarse.

Campo prioridad

Valores baja, media, alta. Por defecto media en creación por técnico/departamento. Define el color/urgencia en la interfaz.

Campo tipo_formato

Valores a, b, c, d. Determina a qué formulario de formato se redirige al completar el ticket y qué tabla de detalle se usa.

Campo tipo_atencion y num_memo (formatos)

tipo_atencion indica cómo se solicitó el servicio: Memo, Teléfono, Jefe o Usuario. Cuando es Memo, se habilita y exige el campo num_memo (número/folio del memorándum de referencia).

13 Flujo de datos y servicios

Recorrido ticket → servicio → formato

  1. Creación — TicketService::crearComo{Admin|Usuario|Departamento}() genera el folio, guarda el ticket en estado nuevo y dispara TicketCreadoMail.
  2. Asignación / toma — asignarComoAdmin() o tomarComoUsuario() pasan el ticket a asignado. La toma usa un update condicional para evitar carreras entre técnicos.
  3. Inicio de atención — al "Completar", iniciarAtencionYCrearServicioSiFalta() crea el servicio (folio institucional) y pasa el ticket a en_proceso.
  4. Llenado de formato — FormatoController::formato{A..D}() muestra el formulario, precargando departamento, solicitante y descripción del ticket.
  5. Cierre — store{A..D}() guarda el detalle y llama a ServicioService::completarTicketPorId(), que marca el ticket como completado y envía TicketCompletadoMail.

Servicios de negocio

TicketService

Creación con folio único, asignación, toma, cancelación, edición por rol y notificaciones de creado/asignado.

ServicioService

Alta de servicios con folio global, y cierre del ticket + notificación de completado.

14 Despliegue y mantenimiento

Instalación

  1. Dependencias PHP — composer install.
  2. Dependencias front — npm install.
  3. Entorno — copiar .env, configurar base de datos y correo, php artisan key:generate.
  4. Base de datos — php artisan migrate (y seeders de roles/permisos).
  5. Ejecución — composer run dev levanta servidor, colas, logs y Vite.
Notas de mantenimiento

Las rutas /test-mail son de prueba de correo y deben deshabilitarse en producción. Verifica la configuración SMTP en .env para que lleguen las notificaciones.

Zona horaria

El sistema opera en America/Mexico_City. Las fechas visibles se formatean a esa zona.
